# Writing a High-Quality PR

A high-quality PR is a gift to maintainers. It's easy to review, easy to merge, and easy to maintain.

## The Ideal PR

```mermaid
pie title "What Maintainers Look For"
    "Clear description" : 25

    "Tests included" : 20
    "Clean code" : 15
    "Good commit messages" : 10

```

## PR Anatomy

```mermaid
flowchart TD
    A[Pull Request] --> B[Title]
    A --> C[Description]
    A --> D[Changes]
    A --> E[Tests]
    A --> F[Documentation]
    
    B --> B1[Clear, concise summary]
    C --> C1[What, Why, How]
    D --> D1[Small, focused commits]
    E --> E1[Prove it works]
    F --> F1[Update if needed]
```

## Before You Start

### Pre-PR Checklist

- [ ] Issue exists (or create one)
- [ ] Approach discussed with maintainers
- [ ] Fork is up to date with upstream
- [ ] New branch created from main/master
- [ ] Contributing guidelines read

### Branch Naming

```bash
# Convention: type/description
feature/add-dark-mode
fix/login-crash-special-chars
docs/update-installation-guide
refactor/simplify-auth-logic
```

## Writing the PR Title

### Format

```
type(scope): brief description

# Examples
feat(auth): add password strength indicator
fix(api): handle null response from server
docs(readme): add Windows installation steps
test(utils): add tests for date formatting
refactor(db): simplify connection pooling
```

### Title Types

| Type | When to Use |
|------|------------|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation only |
| `style` | Formatting, no logic change |
| `refactor` | Code restructuring |
| `test` | Adding tests |
| `chore` | Maintenance tasks |
| `perf` | Performance improvements |

## Keeping PRs Small

### Why Size Matters

```mermaid
quadrantChart
    title PR Size vs Review Quality
    x-axis Small --> Large
    y-axis Low Quality Review --> High Quality Review
    quadrant-1 Quick & Thorough
    quadrant-2 Careful but Slow
    quadrant-3 Neglected
    quadrant-4 Superficial
    
    Small Bug Fix: [0.2, 0.9]
    Feature Module: [0.5, 0.6]
    Major Refactor: [0.7, 0.4]
    Complete Rewrite: [0.9, 0.2]
```

### Size Guidelines

| PR Size | Lines Changed | Review Time | Quality |
|---------|---------------|-------------|---------|
| Tiny | < 50 | Minutes | Excellent |
| Small | 50-200 | Hours | Good |
| Medium | 200-500 | Days | Moderate |
| Large | 500+ | Weeks | Poor |

### How to Keep PRs Small

```mermaid
flowchart TD
    A[Large Feature] --> B[Split by layer]
    A --> C[Split by functionality]
    A --> D[Split by steps]
    
    B --> B1[PR 1: Database changes]
    B --> B2[PR 2: API endpoints]
    B --> B3[PR 3: UI components]
    
    C --> C1[PR 1: Core feature]
    C --> C2[PR 2: Enhancement A]
    C --> C3[PR 3: Enhancement B]
    
    D --> D1[PR 1: Add tests for existing]
    D --> D2[PR 2: Refactor needed parts]
    D --> D3[PR 3: Implement feature]
```

## Commit Structure

### One Logical Change Per Commit

```bash
# âŒ Bad: Everything in one commit
git add .
git commit -m "Add feature"

# âœ… Good: Logical commits
git add src/components/Button.tsx
git commit -m "feat(ui): add Button component"

git add src/components/Button.test.tsx
git commit -m "test(ui): add Button component tests"

git add docs/components.md
git commit -m "docs: add Button documentation"
```

### Clean Up Before PR

```bash
# Squash/fixup messy commits
git rebase -i HEAD~5

# Choose commits to squash:
# pick abc123 feat: add user auth
# fixup def456 fix typo
# fixup ghi789 fix another typo
# fixup jkl012 oops forgot file
```

## Self-Review

Before requesting review:

```mermaid
flowchart TD
    A[Ready to Submit?] --> B{Diff clean?}
    B -->|No| C[Remove debug code]
    B -->|Yes| D{Tests pass?}
    D -->|No| E[Fix tests]
    D -->|Yes| F{Docs updated?}
    F -->|No| G[Update docs]
    F -->|Yes| H{Commits clean?}
    H -->|No| I[Rebase/squash]
    H -->|Yes| J[Submit PR!]
```

### Self-Review Checklist

- [ ] No console.log or debug statements
- [ ] No commented-out code
- [ ] No unrelated changes
- [ ] No merge conflicts
- [ ] Tests pass locally
- [ ] Linting passes
- [ ] Types are correct (if TypeScript)

## Draft PRs

### When to Use Drafts

- Work in progress, need feedback
- Sharing approach before finishing
- CI setup to run on PR
- Blocking on external dependency

## Linking Issues

### How to Link

```markdown
# Keywords that close issues when PR is merged:
Fixes #123
Closes #123
Resolves #123

# Reference without closing:
Related to #123
See #123
Part of #123
```

### Multiple Issues

```markdown
## Related Issues

Fixes #123 - Main bug this addresses
Fixes #124 - Related issue also fixed
Related to #100 - Parent epic
```

## Requesting Review

### When to Request

```mermaid
flowchart TD
    A[PR Created] --> B{Is it a draft?}
    B -->|Yes| C[Wait until ready]
    B -->|No| D{CI passing?}
    D -->|No| E[Fix CI first]
    D -->|Yes| F{Self-reviewed?}
    F -->|No| G[Review your own code]
    F -->|Yes| H[Request review]
```

### How to Request

```markdown
@maintainer - This PR is ready for review. 
It implements #123 as discussed. 
Happy to make any changes you suggest.
```
