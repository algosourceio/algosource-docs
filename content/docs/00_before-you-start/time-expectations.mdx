## The Contribution Timeline

What to expect at each stage:

<Steps>
<Step number={1} title="Month 1: Learning">
**Time:** 10-20 hours total

You're mostly reading:
- Project documentation
- Codebase structure
- Past issues and PRs
- Contribution guidelines

**Output:** Maybe 1 small PR (typo fix, doc improvement)
</Step>

<Step number={2} title="Months 2-3: First Real Contributions">
**Time:** 5-10 hours/week

You're starting to contribute:
- Your first "real" PR
- Responding to review feedback
- Learning project conventions
- Building relationships

**Output:** 1-2 small PRs merged
</Step>

<Step number={3} title="Months 4-6: Building Momentum">
**Time:** 5-10 hours/week

You're becoming a known contributor:
- Tackling bigger issues
- Faster review cycles (maintainers know you)
- Helping other newcomers
- Understanding project priorities

**Output:** 3-5 PRs merged, maybe a feature
</Step>

<Step number={4} title="Months 6-12: Established Contributor">
**Time:** 5-15 hours/week

You're a recognized community member:
- Assigned to issues directly
- Reviewing other people's PRs
- Participating in roadmap discussions
- Mentoring newcomers

**Output:** Regular contributions, recognition
</Step>

<Step number={5} title="Year 2+: Core Contributor / Maintainer">
**Time:** 10-20+ hours/week

You're shaping the project:
- Triage rights or write access
- Release management
- Architecture decisions
- Community leadership

**Output:** Project depends on you
</Step>
</Steps>

### 2. Context Switching

Every time you come back to a PR after days/weeks, you need to:

- Re-read your code
- Remember the discussion
- Understand new feedback
- Rebase/resolve conflicts

<Callout type="tip" title="Pro Tip">
Keep notes on each PR. When you step away, write down where you are and what's next. Your future self will thank you.
</Callout>

### 3. Communication

More time than you expect goes to:

| Activity | Time | Frequency |
|----------|------|-----------|
| Writing detailed PR descriptions | 30-60 min | Every PR |
| Responding to review comments | 15-30 min | Multiple times |
| Asking clarifying questions | 15-30 min | As needed |
| Participating in issue discussions | 30-60 min | Weekly |
| Reading project announcements | 15-30 min | Weekly |

### 4. Tool Setup & Debugging

- Setting up the development environment: 2-8 hours
- Debugging CI failures: 30 min - 2 hours per PR
- Keeping your fork in sync: 15 min weekly
- Learning project-specific tools: 2-5 hours initially

## Time Management Strategies

<CardGrid>
<Card title="Batch Your Work" icon={<Clock className="h-5 w-5" />}>
Set aside 2-3 hour blocks instead of 30-minute snippets. Deep work is more effective.
</Card>

<Card title="Use Dead Time" icon={<Clock className="h-5 w-5" />}>
Code review, issue triage, and documentation can be done on mobile during commutes.
</Card>

<Card title="Accept Wait Times" icon={<Calendar className="h-5 w-5" />}>
Don't let waiting block you. Work on multiple PRs or projects in parallel.
